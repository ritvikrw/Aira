Core design principle: RECEP is a single deployment that serves multiple clients. Client isolation is achieved through a client_id field in every database table and a separate ChromaDB collection per client. There is no separate server, container, or database per client.

Multi-tenancy in the call setup: The LiveKit room name is formatted as client-{slug}-{uuid}. The worker extracts the slug (e.g. apollo), runs SELECT * FROM clients WHERE slug = 'apollo', and loads that client's API keys and agent settings. All API calls during the call use that client's own keys — no shared quota.
Transcript is non-blocking: The transcript write to PostgreSQL is fired as a background async task (asyncio.create_task). It does not add latency to the conversation turn.

Section 5

Storage Requirements

Volume / Path Pilot size Growth rate Contents
/data/postgres 10 – 20 GB ~3–5 GB / month per active client Call logs, full transcripts (every utterance), call summaries, agent settings, client config + API keys.
/data/chroma 2 – 10 GB ~500 MB per 1000 documents Vector embeddings for all client knowledge bases. One ChromaDB collection per client (kb_{client_id}).
/data/uploads 5 – 20 GB Depends on client KB activity Raw files uploaded by clients (PDF, DOCX, TXT). Processed into ChromaDB — originals can be archived to object storage after embedding.
/data/livekit < 1 GB Static LiveKit config and room state only. Audio is not recorded — it streams live and is discarded after the call.
OS + app code ~8 – 12 GB Static Ubuntu OS, Docker images, Node modules, Python packages, application code.
Recommended disk layout: OS disk (80 GB SSD for OS + app) + separate data volume (50 GB at launch, expandable). Mount /data on the separate volume. This way you can resize the data disk without touching the OS.
Backup policy: Run pg_dump daily to a separate location (S3-compatible object storage). ChromaDB has no built-in backup — periodically snapshot /data/chroma using the VM snapshot feature. Uploads directory should be mirrored to object storage in real time.
Section 6

Network & Port Map

Port / Protocol Service Exposure Purpose
80 / TCP Caddy Public HTTP → redirects to HTTPS
443 / TCP Caddy Public HTTPS — entry point for all web traffic. Routes to API / dashboards / LiveKit
7880 / TCP LiveKit Internal / proxied LiveKit HTTP API and WebSocket signalling (proxied by Caddy)
7881 / TCP LiveKit Public WebRTC TCP fallback (when UDP is blocked)
50000–60000 / UDP LiveKit RTP Public Critical. RTP media (audio). Must be open in firewall. LiveKit uses this range for all active call audio.
5060 / UDP+TCP SIP Public SIP signalling from SIP trunk provider. Required for PSTN / phone number calls.
8001 / TCP FastAPI Internal / proxied REST API — proxied by Caddy. Not directly accessible from public internet.
3002 / TCP Client dashboard Internal / proxied Next.js client UI — proxied by Caddy
3003 / TCP Admin dashboard Internal / proxied Next.js admin UI — proxied by Caddy
5432 / TCP PostgreSQL Internal only Never expose to public internet. Only accessible within the VM.
8000 / TCP ChromaDB Internal only Never expose to public internet. Only accessed by FastAPI and voice agent.
Firewall rules summary: Open inbound 80, 443, 7881, 50000–60000 (UDP), 5060. Block everything else including 5432 and 8000. The UDP range 50000–60000 is mandatory for voice to work — if this is blocked, calls will fail or fall back to TCP with high latency.

Scaling trigger — voice agent worker: The worker is the first bottleneck. Each Python process handles ~5 concurrent calls. At Tier 1, run 2 worker processes (10 concurrent calls). At Tier 2, run 6 worker processes (30 concurrent calls). Workers are stateless and can be added without downtime.
Audio bandwidth: LiveKit streams audio at ~50–80 kbps per call leg. At 10 concurrent calls that is ~1.5 Mbps sustained. The 5 TB/month included on Vultr is far more than enough — audio does not dominate bandwidth cost.
Section 8

Multi-tenancy — How It Works

Client isolation in PostgreSQL

Every table has a client_id UUID column. Every query from the API is filtered by this column. Client A can never read or write Client B's data. No separate schema or database per client is needed.

Tableclient_id presentWhat it isolates
clientsPrimary key tableCompany name, slug, is_active, all API keys
call_logsFK → clientsEvery call record
transcriptsFK → clientsEvery utterance (user + agent)
call_summariesFK → clientsLLM-generated summaries, categories, action items
agent_settingsComposite PK (client_id, key)Per-client agent config (language, voice, persona)
knowledge_documentsFK → clientsUploaded file metadata
client_usersFK → clientsDashboard login email + bcrypt password

Client isolation in ChromaDB

Each client gets their own ChromaDB collection named kb_{client_id}. The voice agent queries only the collection for the active client. Documents never cross clients.

How the voice agent knows which client is calling

Room name encoding: When a call starts, the LiveKit room is named client-{slug}-{uuid} (e.g. client-apollo-a3f2b1). The voice agent worker receives the room name with the job, splits on -, and extracts the slug. It then runs SELECT * FROM clients WHERE slug = 'apollo' to load that client's full configuration — API keys, agent settings, language preference — before the first audio frame is processed.
